# 功能对标（对照端）— Cursor 可执行流程

## 目标

在 **不丢弃 Orchestra 自有能力**（服务端路径、工作区、Web 客户端等，见 `CLAUDE.md` 与 `docs/superpowers/specs/2026-03-29-orchestra-design.md`）的前提下，使产品行为、契约与 UX **尽可能与对照端（参考用 Tauri 桌面实现）对齐**。单次会话或单次 PR 可能无法完成一轮闭环，按下面步骤 **分轮迭代**。

## 术语

- **对标**：功能与体验与对照端可对照、语义一致（经 HTTP/WebSocket/路径差异可接受）。
- **一轮迭代**：完成 Step 1 → Step 2 → Step 3 的一次循环；未对齐则开启下一轮 Step 1。

## 工作流（必须按顺序执行）

### Step 1 — 差距分析（产出文档）

**输入**：当前代码库、`docs/gap-analysis.md`（若存在）、对照端行为/文档（团队约定的参考实现或用户提供的对照点）。

**必须产出**：在 `dev_doc/` 下新建 **差距与路线图** 文件：

- 命名：`development-roadmap-parity-{迭代轮次}.md`
- `{迭代轮次}`：两位数字递增，例如 `01`、`02`（与已有 `development-roadmap-parity.md` 并存时，新文件用下一号，避免覆盖旧版总览时可选择保留总览、轮次文件写增量）。

**文档内容建议包含**：

1. 本轮范围（要对标的对照端能力子集）。
2. 当前 Orchestra 状态简述。
3. **差距列表**（每条：对照端期望行为 / Orchestra 现状 / 是否保留 Orchestra 特有差异及理由）。
4. **本轮可交付项**（可在一个 PR 内完成的条目）与 **后续轮次 backlog**。
5. 验证方式（手工步骤、已有 `frontend/e2e`、后端 `make test` 等）。

**约束**：分析时明确写出须 **保留的本项目特性**，避免为对齐而删掉 Web/工作区等设计目标。

**索引**：全周期 REQ 总览见 [`development-roadmap-parity.md`](./development-roadmap-parity.md)；各轮增量见 `development-roadmap-parity-NN.md`。

---

### Step 2 — 实现与验证

**输入**：Step 1 产出的路线图文件。

**执行**：

1. 按路线图中的 **本轮可交付项** 改代码（前后端以仓库现有栈为准）。
2. **测试**：跑相关自动化测试；对 UI/终端/WS 行为做与路线图一致的 **手工或 e2e 核对**。
3. **完成标准**：路线图本轮条目均有对应实现，且测试/核对通过后再进入 Step 3。

---

### Step 3 — 对齐判定与收尾

**输入**：Step 2 的改动与测试结果。

**判定**：

- **已对齐（本轮目标范围内）**：将改动 **提交到 git 仓库**（遵循 Conventional Commits）；可在路线图文件中勾选本轮条目或追加简短「本轮结论」。
- **未对齐**：**不要** 为了「先提交」而降低标准；回到 **Step 1**，用 **新的轮次号** 写 `development-roadmap-parity-{迭代轮次}.md`，记录剩余差距与下一批可交付项。

---

## 给 Cursor Agent 的简短指令模板

复制到对话或规则中即可：

```text
按 dev_doc/功能对标流程.md 执行对照端对标：
1) 在 dev_doc/ 写入 development-roadmap-parity-XX.md（XX 为下一轮次），做差距分析并保留 Orchestra 特有特性；
2) 仅实现该文档「本轮可交付项」，并运行相关测试与核对；
3) 对齐则提交；否则新建下一轮路线图，不强行提交未达标部分。
```
